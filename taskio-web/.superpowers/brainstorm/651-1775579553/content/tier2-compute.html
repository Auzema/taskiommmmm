Tier 2: Which compute approach for the backend API?

Your backend is Node.js + Express with Socket.io (WebSockets). This affects cost, ops complexity, and scalability.

A

ECS + AWS Fargate RECOMMENDED

Containerized Node.js app running on serverless containers. Matches the reference architecture exactly.

Pros

  • Persistent WebSocket connections (Socket.io works natively)
  • No cold starts
  • Container image in ECR — reproducible builds
  • Auto-scaling based on CPU/memory
  • ALB handles sticky sessions for WebSockets

Cons

  • Higher baseline cost than Lambda (~$30+/mo minimum)
  • Slightly more ops: Dockerfile, ECR push, task definitions
B

AWS Lambda + API Gateway

Serverless functions — each API route becomes a Lambda. Pay-per-request model.

Pros

  • Nearly zero cost at low traffic
  • No server management at all
  • Built-in scaling to thousands of requests

Cons

  • WebSockets require separate API Gateway WebSocket + Lambda (complex)
  • Cold starts (latency spikes)
  • 15-min execution limit — not great for long-lived connections
  • Requires significant refactor of Express app
C

EC2 Auto Scaling Group

Traditional VMs with an Auto Scaling Group behind the ALB.

Pros

  • Familiar deployment model (SSH, PM2)
  • Full control over OS and runtime
  • Good for predictable, steady-state traffic

Cons

  • You manage OS patching, security updates
  • Slower scaling (minutes vs seconds for Fargate)
  • Less cost-efficient: pay for whole instance 24/7
