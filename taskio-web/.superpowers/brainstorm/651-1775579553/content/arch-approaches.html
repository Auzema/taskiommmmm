3 Architecture Approaches for Tier 2 Entry Point

Tier 1 (S3 + CloudFront + Route 53) and Tier 3 (DynamoDB + ElastiCache) are fixed. The difference is how traffic reaches ECS Fargate.

A

ALB-Direct + WAF RECOMMENDED

Route 53 → CloudFront (frontend) / ALB (API) → ECS Fargate → DynamoDB + ElastiCache

WAF attached to CloudFront and ALB. Cognito Authorizer on ALB listener rules. Clean and proven pattern for containerized apps.

Pros

  • Native WebSocket support via ALB + sticky sessions
  • WAF at edge (CloudFront) + at ALB for defense-in-depth
  • Lower latency — one fewer hop vs API Gateway
  • Cognito JWT validation built into ALB listener rules
  • Lower cost than API Gateway at scale

Cons

  • No built-in request/response transformation
  • API throttling requires custom logic or WAF rate rules
B

API Gateway + ALB (Full Reference Arch)

Route 53 → CloudFront / API Gateway → ALB → ECS Fargate → DynamoDB + ElastiCache

Matches the reference diagram exactly. API Gateway provides throttling, caching, and Cognito Authorizer. ALB routes to Fargate tasks.

Pros

  • Native Cognito Authorizer at API Gateway level
  • Built-in request throttling and usage plans
  • API response caching
  • Looks great on architecture diagrams

Cons

  • WebSockets need separate API Gateway WebSocket API (more complex)
  • Higher cost per request at scale
  • Two hops before reaching Fargate (latency)
  • HTTP API GW → ALB → Fargate is more complex to configure
C

CloudFront as Unified Proxy

Route 53 → CloudFront (frontend + API proxy) → ALB → ECS Fargate → DynamoDB + ElastiCache

Single CloudFront distribution handles both static assets (S3 origin) and API calls (ALB origin). Path-based routing: /v1/* → ALB, /* → S3.

Pros

  • Single domain for frontend + API (no CORS issues)
  • Edge caching for API responses where applicable
  • Simplified DNS setup

Cons

  • WebSocket connections over CloudFront require careful configuration
  • CloudFront timeouts can disrupt long-lived connections
  • More complex CloudFront behavior rules
