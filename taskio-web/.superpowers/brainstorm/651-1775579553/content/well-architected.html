☁️ AWS Well-Architected Framework — Taskio Compliance

How each design decision maps to the 6 pillars. Use this section in your presentation and defense.

⚙️
1. Operational Excellence
Run, monitor, and continuously improve operations
CodePipeline + CodeBuild
Fully automated CI/CD — every git push triggers build, test, and ECS rolling deploy with zero manual steps and zero downtime.
CloudWatch Logs & Alarms
All ECS task stdout/stderr streamed to CloudWatch. Alarms on CPU > 80%, ALB 5xx rate, DynamoDB throttling — ops team is alerted before users notice.
ECS Fargate
No OS patching, no AMI management, no SSH sessions. AWS handles the runtime — our team focuses on application logic, not infrastructure maintenance.
Secrets Manager
Centralized config management — no hardcoded env vars in code or Dockerfiles. Secrets rotate automatically without redeploying.
ECS Rolling Deploy
Health-check gated rollout (min 50% healthy). Failed deploys auto-rollback before old tasks are terminated — safe by default.
🔐
2. Security
Protect data, systems, and assets at every layer
Amazon Cognito
Managed auth — eliminates custom JWT signing, bcrypt misuse, and token rotation bugs from our codebase. SOC2/ISO-certified identity provider.
WAF on ALB
OWASP Top 10 managed rule set blocks SQLi, XSS, and HTTP floods before requests reach ECS. IP-based rate limiting (1000 req/5min) prevents abuse.
VPC Private Subnets
ECS tasks and ElastiCache are in private subnets — zero inbound internet access. ALB is the only public entry point to the application tier.
ACM + HTTPS
TLS 1.2+ enforced on CloudFront and ALB. ACM auto-renews certificates — no manual cert management.
DynamoDB VPC Endpoint
Database traffic never traverses the public internet. VPC Gateway Endpoint routes DynamoDB calls within AWS's private backbone network.
IAM Least Privilege
ECS task execution role grants only: ECR pull, Secrets Manager read, CloudWatch write. No wildcard permissions.
🛡️
3. Reliability
Recover from failures and meet demand automatically
Multi-AZ (AZ1 + AZ2)
ECS tasks, ElastiCache, and ALB targets span 2 AZs. If an entire AZ goes down, ALB automatically routes all traffic to the healthy AZ within seconds — no manual intervention.
ALB Health Checks
GET /v1/status checked every 30s. Unhealthy ECS tasks are deregistered automatically — traffic never hits a broken instance.
DynamoDB
Built-in Multi-AZ replication with 99.999% availability SLA. On-demand capacity means no throttling during traffic spikes.
ElastiCache Primary + Replica
Primary in AZ1, replica in AZ2. Automatic failover in ~30s if primary fails — cache stays available during AZ failure.
ECS Auto Scaling
Target tracking on CPU utilization — new Fargate tasks spin up in ~60s when load increases. Min 2 tasks (one per AZ) ensures baseline HA at all times.
NAT Gateway × 2
One NAT Gateway per AZ. If AZ1 NAT fails, AZ2 Fargate tasks still have outbound internet access via AZ2 NAT — no single point of failure for egress.
⚡
4. Performance Efficiency
Use resources efficiently as demand changes
CloudFront
React bundle served from 400+ edge PoPs worldwide — users load the app from the nearest location, not your origin region. Sub-10ms asset delivery globally.
ElastiCache (Redis)
Sub-millisecond cache layer absorbs repetitive DynamoDB reads (user sessions, board permissions, workspace data). Cache hit = no DB round-trip.
DynamoDB On-Demand
No provisioned capacity limits — handles sudden traffic spikes (viral board shares, large team onboarding) without pre-warming or throttling.
ECS Fargate Task Sizing
Right-size per task (512 vCPU / 1GB RAM baseline), not per VM. Scale individual tasks without over-provisioning idle instance capacity.
Socket.io Redis Adapter
Real-time events broadcast via Redis pub/sub — O(1) delivery to any connected client regardless of which Fargate task they're on. No polling overhead.
ALB Connection Reuse
ALB maintains persistent connections to ECS targets, reducing TCP handshake overhead for high-frequency API calls from the React frontend.
💰
5. Cost Optimization
Deliver value at the lowest price point
ECS Fargate
Pay per task-second — no idle EC2 costs. Scale to zero during off-hours if needed. No reserved instance commitments required to start.
DynamoDB On-Demand
Pay per read/write request — no minimum hourly charge. Ideal for variable traffic patterns. Switch to provisioned + auto-scaling at sustained high load for further savings.
S3 + CloudFront (Frontend)
Hosting a React SPA on S3+CloudFront costs cents per month. Eliminates any server cost for the presentation tier entirely.
ElastiCache → DynamoDB reduction
Each cache hit avoids a DynamoDB read. At 1M cached reads/day, savings are ~$0.13/day. At scale, the cache pays for itself by reducing DB read costs.
DynamoDB VPC Endpoint
Free to create. Eliminates NAT Gateway data processing fees for all DynamoDB traffic — potentially saving $0.045/GB on high-volume DB workloads.
Single WAF on ALB
WAF only on the API entry point (ALB) — not duplicated on CloudFront for static assets. Avoids redundant WAF cost (~$5/mo + per-request fee) on low-risk S3 traffic.
🌱
6. Sustainability
Minimize environmental impact of cloud workloads
ECS Fargate (Shared Fleet)
AWS bins Fargate tasks onto shared, highly-utilized physical servers — far higher hardware utilization than a dedicated EC2 instance. Less energy per compute unit.
Auto Scaling (No Idle Waste)
Tasks scale to match demand. No over-provisioned servers running at 5% CPU at 3am. Only the compute actually needed is consuming energy.
DynamoDB + ElastiCache (Managed)
AWS manages the hardware fleet at massive scale — data center PUE and server utilization rates are far better than any self-hosted MongoDB/Redis cluster.
CloudFront Edge Caching
Cached responses at edge PoPs prevent repeat origin fetches — fewer compute cycles per user request. Each cache hit = zero server processing energy.
S3 Static Hosting
S3 stores objects across a shared, AWS-optimized storage fleet. No dedicated server running for your frontend — effectively zero marginal energy cost per request.
AWS Region Selection
Deploying in an AWS region powered by renewable energy (e.g., ap-southeast-1 has renewable commitments) further reduces the workload's carbon footprint.
