📦 What Each Component Does & How They Work Together
🔵Route 53Tier 1
DNS + health-based routing
Resolves taskio.com → CloudFront (frontend) and api.taskio.com → ALB. Performs health checks; if ALB goes unhealthy, Route 53 stops routing traffic to it.
⚡CloudFrontTier 1
Global CDN for the React app
Caches the S3 static build at 400+ edge nodes worldwide. Users load JS/CSS from the nearest PoP, not your S3 region. Also terminates HTTPS (ACM cert). WAF is attached here as the first security gate.
🛡️WAFTier 1NEW
Web Application Firewall
Attached to both CloudFront and ALB (defense-in-depth). Blocks OWASP Top 10 (SQLi, XSS), rate-limits by IP (e.g., 1000 req/5min), and can geo-block. CloudFront WAF stops attacks at edge before they hit your VPC.
🪣S3 (Frontend)Tier 1KEPT
Static asset hosting
Stores the vite build output. Bucket is private — only accessible via CloudFront OAC (Origin Access Control). npm run build → upload to S3 → CloudFront invalidation.
👤Amazon CognitoTier 1NEW
Managed user auth (replaces custom JWT)
Handles registration, email verification, password reset, and token issuance. Issues JWTs (AccessToken, IdToken, RefreshToken). Frontend calls Cognito SDK directly — no auth traffic hits your backend. ALB verifies the JWT signature using Cognito's JWKS endpoint automatically.
Replaces: bcryptjs, JwtProvider, verifyToken flow, Brevo email for auth
⚖️ALBTier 2
L7 load balancer + Cognito enforcer
Sits in public subnets, receives HTTPS from Route 53. Listener rules validate Cognito JWT before forwarding to Fargate — unauthenticated requests are rejected at the ALB, never reaching your app. Sticky sessions (duration-based) ensure WebSocket clients reconnect to the same Fargate task.
📦ECS FargateTier 2
Serverless containers running Node.js API
Runs your Express + Socket.io app in Docker containers. AWS manages the underlying EC2 — you only define CPU/memory per task. Auto Scaling adds tasks when CPU > 70%. In AZ1 and AZ2 for HA. Tasks pull env vars from Secrets Manager at startup.
🌐NAT GatewayTier 2
Outbound internet for private subnets
Fargate tasks in private subnets need outbound internet (Cloudinary, S3 public API, SES). NAT Gateway (one per AZ) provides this without exposing tasks to inbound internet traffic. Each costs ~$32/mo + data transfer.
🗄️DynamoDBTier 3↔ MongoDB
Primary NoSQL database
Multi-table design: one DynamoDB table per collection (22 tables). Each entity uses its MongoDB _id equivalent as partition key. GSIs (Global Secondary Indexes) replace MongoDB's $lookup queries. On-demand capacity — scales automatically, pay per request. Accessed via VPC Endpoint (no internet hop).
⚡ElastiCache RedisTier 3↔ Redis
Cache + Socket.io pub/sub broker
Two roles: (1) API response caching — user sessions, board data, permissions cached with TTL to reduce DynamoDB reads. (2) Socket.io Redis Adapter — when multiple Fargate tasks run, Socket.io events are broadcast across all instances via Redis pub/sub. Primary in AZ1, replica in AZ2 for HA failover.
📦ECRCI/CD
Docker container registry
Stores your Docker images. CodeBuild pushes a new image tagged with the Git commit SHA. ECS task definitions reference the ECR image URI. Private registry — only your VPC and CodeBuild IAM role can pull.
🔨CodeBuild + CodePipelineCI/CDNEW
Build and deploy automation
CodePipeline: GitHub push → Source stage → Build stage (CodeBuild) → Deploy stage (ECS rolling update). CodeBuild: runs npm run build, builds Docker image, pushes to ECR, writes imagedefinitions.json. ECS rolling update replaces tasks one by one — zero downtime deployment.
👁️CloudWatchSupport
Observability (logs + metrics + alarms)
ECS tasks stream logs to CloudWatch Logs. Alarms on: CPU > 80% (trigger ECS scale-out), DynamoDB throttling, ElastiCache evictions, ALB 5xx rate. Dashboards for request count, latency, error rate.
🔑Secrets ManagerSupport
Secure env var storage
Stores CLOUDINARY_KEY, SES_SMTP, REDIS_PASSWORD, and other secrets. ECS task definition references Secrets Manager ARNs — injected as env vars at container start. Automatic rotation for supported secrets.

🟢 Tier 1 — Defense Questions
Why S3 + CloudFront instead of hosting the frontend on EC2/ECS?
React compiles to static files — there is no reason to run a server for it. S3 + CloudFront is the AWS best practice for static SPAs: zero server management, sub-millisecond edge delivery, 99.99% availability SLA, and near-zero cost at our scale. Running EC2 for static files wastes compute budget and adds unnecessary failure points.
Why Cognito instead of keeping your custom JWT system?
Custom auth means we own every security vulnerability: token rotation bugs, brute-force attack surface, PKCE implementation, MFA complexity. Cognito is SOC2/ISO-certified, handles all of that, and integrates natively with ALB — the ALB verifies Cognito JWTs automatically using JWKS, removing auth logic entirely from our app. It also gives us MFA, social login, and Cognito-triggered SES emails for free.
Caveat
Migration requires refactoring auth routes and frontend auth flows. Cognito has limited customization (hosted UI is basic). Vendor lock-in to AWS.
Defense
The frontend uses its own UI (not Cognito hosted UI) — only the token exchange changes. Lock-in is acceptable because our entire stack targets AWS. Migration effort is bounded and one-time.
Why WAF on both CloudFront AND ALB?
Defense-in-depth. CloudFront WAF stops attacks at the edge before they consume VPC resources or reach the ALB. ALB WAF is a backstop for requests that enter VPC through other paths. This is the AWS Well-Architected Framework security best practice for public-facing web apps.
🔵 Tier 2 — Defense Questions
Why ECS Fargate and not AWS Lambda?
Our app uses Socket.io for real-time board invitations — this requires persistent, long-lived TCP connections. Lambda functions are stateless and terminate after execution; they cannot maintain WebSocket connections without a separate API Gateway WebSocket API, which would require significant refactoring. Fargate runs our existing Express app in a container with zero code changes to the WebSocket layer.
Caveat
Fargate has a higher baseline cost (~$30+/mo minimum per task) versus Lambda's pay-per-invocation model at low traffic.
Defense
For a real-time collaborative app, Fargate's always-on model is correct — users expect instant responses. Lambda cold starts (100ms–3s) would degrade the experience. The cost is justified by the latency requirements.
Why ALB and not API Gateway in front of ECS?
ALB natively supports WebSocket upgrades and sticky sessions — API Gateway HTTP API can forward to ALB but WebSockets require a separate API Gateway WebSocket API, adding complexity and cost. ALB also has a Cognito OIDC authenticator built in, so JWT validation happens before the request ever hits ECS. Lower latency (one fewer hop) and lower cost per request at scale.
How does Socket.io work with multiple Fargate tasks?
Without coordination, two Fargate tasks each have their own in-memory Socket.io rooms — a client connected to Task A won't receive events emitted by Task B. We solve this with the socket.io-redis-adapter package: all Socket.io events are published/subscribed through ElastiCache Redis, so any task can broadcast to any connected client regardless of which task they're connected to. ALB sticky sessions ensure a client's WebSocket upgrade stays on one task for the connection lifetime.
Caveat
Requires adding @socket.io/redis-adapter to the backend — a small but required code change.
Why Multi-AZ? Isn't it just double the cost?
Multi-AZ provides automatic failover if an entire AWS Availability Zone goes down — a real event that has happened historically. With a single AZ, an AZ outage takes our entire app offline. With two AZs, the ALB continues routing to the healthy AZ within seconds. For a production SaaS serving paying customers, this is the minimum viable HA configuration. The cost increase is primarily two NAT Gateways (~$64/mo combined) — a small price for 99.99% uptime SLA.
🟣 Tier 3 — Defense Questions
Why DynamoDB instead of keeping MongoDB Atlas?
DynamoDB is fully AWS-native: no cluster to manage, automatic Multi-AZ replication, point-in-time recovery, and on-demand scaling with zero provisioning. MongoDB Atlas is a third-party service that requires separate billing, VPC peering setup, and its own availability SLA chain. DynamoDB integrates natively with IAM, VPC Endpoints (no internet traffic), and CloudWatch. For an AWS-hosted app, DynamoDB removes an external dependency.
Caveat — This is the hardest migration
DynamoDB has no joins. Our app uses MongoDB $lookup aggregations heavily (e.g., fetching board + columns + cards in one query). DynamoDB requires redesigning access patterns using GSIs (Global Secondary Indexes) or application-side joins. ObjectIds must be mapped to string keys. This migration is non-trivial.
Defense
We use a multi-table design (one table per collection) which mirrors our existing MongoDB structure and is the most maintainable approach. GSIs cover our main query patterns (boardId, workspaceId, userId lookups). AWS DMS (Database Migration Service) can handle the initial data migration from Atlas to DynamoDB.
What does ElastiCache do beyond what Redis already does?
Our existing Redis is self-managed (wherever it runs now). ElastiCache is a managed Redis cluster: automated failover to the replica in AZ2 (within ~30s), automated backups, patch management, and CloudWatch metrics out of the box. The application code change is minimal — just updating the Redis connection endpoint. The key addition is the Socket.io Redis Adapter use case described above.
Why a DynamoDB VPC Endpoint?
Without a VPC Endpoint, DynamoDB traffic from Fargate tasks would route through the NAT Gateway — adding latency and NAT data transfer costs. A Gateway VPC Endpoint for DynamoDB routes traffic directly within AWS's private network, bypassing the NAT Gateway entirely. It's free to create and reduces both cost and latency.
🟡 CI/CD — Defense Questions
Why CodePipeline over GitHub Actions?
CodePipeline is AWS-native: it integrates directly with ECR, ECS, and IAM without managing AWS credentials as GitHub secrets. ECS rolling deployment is a first-class CodePipeline action — no scripting needed. For an architecture presentation targeting AWS, using AWS-native CI/CD is consistent and demonstrates full AWS stack proficiency.
Caveat
CodePipeline is less flexible than GitHub Actions for complex workflows (matrix builds, test parallelism). It also costs $1/pipeline/month.
Defense
Our pipeline is straightforward: build → push → deploy. CodePipeline handles this elegantly. The cost is negligible.
What is a rolling deployment and why use it?
ECS rolling update replaces Fargate tasks one at a time: spin up a new task with the new image → health check passes → terminate one old task → repeat. At no point are all tasks offline simultaneously. This gives zero-downtime deployments. We can configure minimum healthy percent (e.g., 50%) and maximum percent (e.g., 200%) to control the pace.
⚠️ Known Caveats — Own Them Before They Ask
Caveat 1 — DynamoDB Migration Complexity
MongoDB uses flexible $lookup aggregations. DynamoDB requires pre-planned GSIs for every query pattern. Under-designed GSIs = expensive full-table scans.
Our Answer
We will conduct an access pattern analysis before schema design. Each collection's top 5 query patterns will map to a partition key + GSI combination. We use multi-table (not single-table) design for maintainability. AWS DMS handles the initial data migration.
Caveat 2 — Cognito Auth Refactor Effort
The existing auth system (register, verify, reset, refresh token) is tightly coupled throughout the backend. Replacing it with Cognito requires updating auth routes, middleware, and frontend SDK integration.
Our Answer
The scope is bounded: auth.middleware.js switches to Cognito JWKS verification, auth routes (/register, /login, /verify) are removed and replaced by Cognito SDK calls on the frontend, and email flows are handled by Cognito + SES triggers. The rest of the 12 API route groups are untouched.
Caveat 3 — Socket.io Redis Adapter Required
Multi-task Fargate requires adding @socket.io/redis-adapter. Without it, real-time events only reach clients on the same task — silent failure under load.
Our Answer
This is a known, well-documented pattern. Adding the adapter is 10 lines of code in server.js. ElastiCache is already in the architecture specifically to support this.
Caveat 4 — NAT Gateway Cost
Two NAT Gateways (one per AZ) cost ~$32/mo each plus data transfer fees (~$0.045/GB). For low-traffic MVPs this can be disproportionate.
Our Answer
We target Medium scale / production-grade (our stated requirement). NAT is required for Fargate private subnet outbound access. The cost is fixed and predictable. At scale, data transfer costs are a fraction of overall infrastructure spend. Alternatively, VPC Endpoints for commonly used AWS services (S3, DynamoDB, ECR) reduce NAT data transfer costs significantly.
Caveat 5 — No API Gateway = No Built-in Throttling
Choosing ALB over API Gateway means we don't get per-route throttling or usage plans natively.
Our Answer
WAF rate-based rules on the ALB provide IP-level rate limiting (e.g., block IP after 1000 req/5min). Application-level throttling (per-user quotas) can be implemented in the Express middleware using the existing Redis connection — a standard pattern. This is sufficient for our scale tier.
🗺️ Diagram Q&A — Questions From the Presentation Session
These came up during our design session. Know them cold — they will come up again.
Why are there 2 WAFs in the diagram?
They are on two completely separate traffic paths — not redundant, not the same WAF appearing twice.

• WAF #1 on CloudFront → protects taskio.com (the React frontend served from S3). Stops DDoS and hotlinking at the edge before requests even enter your AWS region.
• WAF #2 on ALB → protects api.taskio.com (your Node.js API). Enforces OWASP Top 10 rules, rate limits per IP, blocks malformed requests before they hit ECS Fargate.

They have no connection to each other. The dashed line between them in older versions of the diagram was a mistake and has been removed.
Follow-up: Could you use just one WAF?
Yes — attach WAF only to the ALB. The real attack surface is the API, not static S3 files. Removing the CloudFront WAF saves ~$5-10/mo and reduces complexity. It's a valid cost trade-off for early-stage deployments.
The app is a monolith, not microservices — is ECS + Fargate still the right choice?
Yes. ECS Fargate runs containers, not microservices. Microservices is an application architecture pattern — ECS is just a container orchestrator that doesn't care whether your container holds one service or twenty.

What actually happens: the entire Express monolith is packaged as one Docker image, ECS runs N identical copies of it behind the ALB. That is horizontal scaling of a monolith — completely standard and production-proven.

The only things that would break monolith horizontal scaling are: (1) in-memory state not shared across instances → solved by ElastiCache Redis, (2) WebSocket rooms not shared across instances → solved by Socket.io Redis Adapter. Both are already in the architecture.
Follow-up: What about Elastic Beanstalk or App Runner?
App Runner has no WebSocket support — Socket.io breaks. Elastic Beanstalk uses managed EC2 under the hood with more ops overhead than Fargate. Fargate is the cleanest choice for a containerized monolith with real-time requirements.
What is ECS and what is ECR? Why are both needed?
• ECR (Elastic Container Registry) = private warehouse for Docker images. Like Docker Hub but hosted inside AWS, with IAM access control. Stores your built image tagged by Git commit SHA (e.g., taskio-api:abc123).
• ECS (Elastic Container Service) = the orchestrator that pulls images from ECR and runs them as Fargate tasks. Handles health checks, auto-scaling, rolling deploys, and placement across AZs.

They're separate because storage and execution are separate concerns. ECR is the "what to run". ECS is "how and where to run it". The deploy pipeline is: CodeBuild builds image → pushes to ECR → ECS task definition references the ECR image URI → Fargate pulls and runs it.
What does "VPC EP" mean on DynamoDB?
VPC EP = VPC Gateway Endpoint. Without it, traffic from ECS Fargate (in a private subnet) to DynamoDB would route: private subnet → NAT Gateway → public internet → DynamoDB. You pay NAT data transfer fees ($0.045/GB) on every database call.

With the Gateway Endpoint, traffic routes directly over AWS's private backbone network — no internet, no NAT, no data transfer cost. It is free to create, requires zero application code changes, and is faster (one fewer network hop). A pure win on cost, performance, and security.
What do the dashed lines mean vs solid lines in the diagram?
• Solid lines = live request/response traffic — actual user data flowing through the system (e.g., browser → Route53 → ALB → ECS → DynamoDB).
• Dashed lines = configuration, credentials, or side-channel relationships — not user traffic.

Specific dashed lines explained:
• ACM → CloudFront / ALB: ACM provisions the TLS certificate onto those services once at setup time. Not in the request path at runtime.
• Cognito → ALB ("JWT validation"): ALB calls Cognito's JWKS endpoint in the background to verify token signatures. No user request is forwarded to Cognito.
• Fargate → CloudWatch: Container logs are streamed as a side-channel. No user request touches CloudWatch.
• Fargate → Secrets Manager: At container startup only — the task fetches env vars before serving any traffic.
• ECR → Fargate ("pull image"): Happens only during deployment, not during user requests.
• ElastiCache replication: Internal Redis sync between primary (AZ1) and replica (AZ2) — invisible to users.
Why don't we use EC2 directly?
ECS Fargate IS running on EC2 under the hood — AWS manages the underlying instances invisibly. The difference is who owns the operational burden:

• Raw EC2: you provision instances, choose AMIs, patch OS, manage SSH keys, set up auto-scaling launch templates.
• ECS on EC2: you still manage the EC2 fleet, just add container orchestration on top.
• ECS Fargate (our choice): you define task CPU/memory, AWS handles everything else. No AMIs, no SSH, no patching, no capacity management.

For a containerized Node.js app, Fargate eliminates an entire class of operational work with zero trade-off in capability.
Should we use Amazon MemoryDB instead of ElastiCache for Redis?
ElastiCache is the correct choice for our use case. The distinction is:

• ElastiCache = in-memory cache. Data loss on failure is acceptable because data lives in DynamoDB (the source of truth). Used for: TTL-based API response caching, Socket.io pub/sub adapter.
• MemoryDB = durable Redis-compatible database with Multi-AZ transaction log. Pay ~2x more for durability guarantees.

In our architecture, DynamoDB is the primary store. Redis is purely complementary — a cache and a pub/sub bus. Cache misses just fall through to DynamoDB. MemoryDB is the right choice only when Redis itself is the durable source of truth (e.g., replacing DynamoDB entirely with Redis). Using it here would mean paying database-grade durability costs on a cache, which a solution architect panel would question.
